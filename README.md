# QR Ticket Validation — Backend

Бэкенд для валидации билетов на мероприятия по QR-коду на входе. Сделан как учебный/портфолио-проект с упором на чистую слоистую архитектуру, миграции БД и корректную обработку реальных пограничных случаев (повторные сканирования, офлайн-синхронизация).

> 🇬🇧 English version of this README is available on request.

## Что делает

Приложение на входе сканирует QR-код билета и отправляет его на этот бэкенд, который отвечает, валиден ли билет, и фиксирует проход. При повторном сканировании того же QR сервер сообщает, что владелец **уже прошёл**, включая время и номер входа.

Бэкенд отвечает на три вопроса:
1. Валиден ли билет? (`/validate`)
2. Зафиксировать проход (`/checkin`)
3. Принять пачку офлайн-проходов (`/sync`)

## Стек

- **Java 17**, **Spring Boot 3.5**
- **PostgreSQL 16**
- **Liquibase** для миграций схемы
- сборка **Maven**, **Lombok**

## Архитектура

Слоистая: `controller → service → repository`. Бизнес-логика живёт в сервисах; сущности наружу не отдаются (вместо них возвращаются DTO).

Три таблицы:
- `tickets` — выпущенные билеты (code как UUID, статус, период действия)
- `checkins` — журнал проходов (одна запись на один зафиксированный вход)
- `gates` — точки входа (заложено под несколько входов)

Ключевые решения:
- **QR = UUID, а не JWT** — статус билета (использован / отменён) всё равно нужно проверять на сервере, поэтому подписанный токен добавил бы сложности без выгоды.
- **Идемпотентность через `client_event_id`** — уникальный id события, генерируемый на устройстве. Если устройство повторно пришлёт тот же проход после потери связи, сервер вернёт прошлый результат вместо создания дубля.
- **`checked_at` (время устройства) ≠ `received_at` (время сервера)** — хранятся раздельно, чтобы офлайн-проходы сохраняли реальное время входа.
- **Схемой владеет Liquibase, Hibernate её валидирует** — миграции единственный источник истины; Hibernate работает в режиме `validate` и никогда не меняет схему сам.

## Локальный запуск

**Требования:** JDK 17, PostgreSQL 16, Maven (или используйте встроенный Maven wrapper).

1. Создайте базу:
   ```sql
   CREATE DATABASE qr_ticket;
   ```
2. (Опционально) задайте креды БД через переменные окружения — иначе используются локальные значения по умолчанию:
   ```
   DB_USERNAME=postgres
   DB_PASSWORD=ваш_пароль
   ```
3. Запустите приложение:
   ```bash
   ./mvnw spring-boot:run
   ```
   Liquibase создаст схему автоматически при первом старте.
4. Проверьте, что поднялось:
   ```
   GET http://localhost:8080/api/v1/health  →  OK
   ```

## API (черновик)

| Метод | Путь | Назначение |
|-------|------|------------|
| GET | `/api/v1/health` | Проверка живости |
| POST | `/api/v1/tickets/validate` | Проверка билета (Спринт 2) |
| POST | `/api/v1/tickets/checkin` | Фиксация прохода (Спринт 2) |
| POST | `/api/v1/sync` | Приём офлайн-проходов (Спринт 3) |

`/validate` и `/checkin` возвращают `result` со значением `OK_TO_ENTER`, `ALREADY_USED`, `NOT_FOUND` или `INVALID`.

## Scope

Входит: валидация билета на входе, фиксация прохода, идемпотентная обработка повторных сканирований, офлайн-синхронизация, утилита генерации тестовых билетов.

Не входит: продажа билетов (внешняя система), мобильное приложение (отдельный трек), генерация реальных QR при продаже.
